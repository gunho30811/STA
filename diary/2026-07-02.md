## 삼삼엠투 통합 채팅 (여러 계정 연결)

한 회원이 삼삼엠투 계정을 여러 개 쓸 때, 임대인 모드 채팅(손님 문의)을 한 곳에서 보게 하는 기능.

- **정찰**: 삼삼엠투 채팅은 REST가 아니라 Firebase RTDB(`spacev-33m2-rdb`)를 웹소켓으로 직접 씀.
  로그인 시 33m2 백엔드가 Firebase 커스텀 토큰 발급 → `signInWithCustomToken`으로 교환한 `refreshToken`만
  있으면, 이후엔 브라우저 없이 `securetoken.googleapis.com`(토큰 갱신) + RTDB REST(`live/chatlist/{uid}`,
  `live/messagelist/{room}`)만으로 폴링 가능함을 확인.
- **스키마**(`db.py`): `samsam_accounts`(회원↔삼삼계정, 비번·refreshToken은 `crypto_util.py`(Fernet)로
  암호화 저장) / `samsam_chat_rooms` / `samsam_chat_messages`. 계정당 여러 개 연결 가능하게 정규화 테이블로.
- **`pipeline/samsam/chat_auth.py`**: Playwright 로그인 1회 → refreshToken 획득. 이후 갱신은 순수 HTTP.
- **`pipeline/samsam/chat_poll.py`**: 계정별 폴링. refreshToken 만료 시 저장된 비번으로 자동 재로그인.
  `host_or_guest` 필드로 임대인 모드 채팅만 저장(게스트로 예약한 채팅은 제외).
- **웹**(`web/samsam_app.py` + `templates/samsam_chat.html`): `/samsam/chat/`에서 계정 연결 폼 +
  통합 인박스(방 목록 + 메시지 스레드). "지금 새로고침" 버튼은 그 회원 계정만 즉시 폴링.
- 로컬에서 실계정으로 연결→폴링→UI까지 end-to-end 확인 후, 테스트로 넣었던 계정은 삭제(다른 회원에게
  안 보이게).
- 다음 단계(미착수): 주기적 폴링용 GH Actions workflow.

## 삼삼 예약률 매일 갱신 "0건" 진단 & 가시성/오염방지 수정 (fix/samsam-refresh-visibility)

- **증상**: 매일 도는 삼삼 크롤 액션이 초록불인데 예약률 갱신이 매번 "0건". DB 교차검증 결과
  07-01 17시 이후 갱신 행 0건 = 저녁 스케줄 실행이 DB에 한 줄도 안 씀(표시버그 아님).
- **오진 → 정정**: 처음엔 "정상 공실(빈 배열)"로 봤으나, 로컬 재현에서 40건 중 17건 정상 데이터가
  왔고(순차·병렬 무관) 코드/ API 자체는 정상임을 확인. 이후 재테스트 중 로컬 IP가 **429(Too Many
  Requests)** 로 막힘 — 1병렬도 429. 즉 **원래 "전건 0"은 정상 공실이 아니라 429 레이트리밋을
  옛 `_get`이 조용히 삼켜(비200→None→빈dict→스킵) "0건"으로 위장**한 것. 통계적으로 10,826건 전건 0은
  정상일 수 없음(정상이면 ~80%가 데이터 있음).
- **수정**(`pipeline/samsam/crawler.py`):
  - `_get(stats=)`: 응답을 `ok/http_<code>/code_<code>/nojson/exc`로 분류·집계(진단 로그).
  - `fetch_schedules` → `(schedules, ok)` 반환. ok=True면 **빈 배열도 '예약 0 확정=공실'로 DB 기록**
    (공실 예약률 0%를 못 남기던 문제 해결), ok=False(차단/에러)면 제외해 기존 값 보존.
  - 리프레시 결과를 모아 검증 후 일괄 커밋 + 성공/데이터有/공실/실패 + HTTP분포 로깅.
  - **오염 방지 가드**: 성공0 / 성공많은데 데이터0 / 초반 200건 데이터0 → 0으로 도배 안 하고
    `exit 1`(빨간불). 다시는 "0건인데 초록불" 위장 불가.
  - `SAMSAM_REFRESH_WORKERS` env로 워커수 조정(기본 2).
- **스코프 확정**: 예약률 갱신 대상 = 서울 전체+경기+인천(현행 유지, 강남구 한정 아님).

### 진짜 원인 규명: 레이트리밋 (실측)
- 로컬 프로브로 정밀 측정: **스케줄 API는 IP당 ~100요청에서 429**, 계속 때리면 안 풀리고
  **때리기를 멈추면 ~15초 내 회복**(rolling window). 즉 원래 "0건"은 매일 10,826건(~38,000요청)을
  병렬로 몰아쳐서(#70 "매일 전체 병렬 갱신") 100요청 벽에 박고 나머지 전부 429로 스킵된 것.
- #70 이전 `a8b8261`의 **"로테이션 갱신"**(하루 일부씩)이 사실 이 한도를 우회하던 설계였는데,
  #70이 "매일 전체 병렬"로 바꾸며 회귀한 것.

### 최종 설계: 로테이션 + 429 백오프 (+ 샤딩 옵션)
- **429 백오프**: 전 워커 공유 쿨다운(`_rl_wait`/`_rl_trip`). 429 만나면 모든 워커가 함께
  멈췄다(창 회복) 재개, `_get`이 `RL_RETRY`회 재시도. (기존엔 429를 그냥 스킵)
- **로테이션**: `collected_at` 오래된 순 정렬 + `SAMSAM_REFRESH_DAILY_LIMIT`(워크플로 5000)로
  하루치만. 매물당 ~2.2초(429 쿨다운 포함)라 전체는 ~397분(타임아웃 330분 초과) → 5,000건씩
  잘라 ~2.2일 주기로 전체 순환.
- **샤딩**(옵션): `SAMSAM_SHARD='i/N'`으로 room_id%N 분할(러너 IP별). IP당/계정당 한도 불확실해
  기본은 미사용, 필요시 matrix로 확장.
- **실시간 증분 커밋**: 데이터 확인 즉시 체크포인트 통과 → BATCH마다 DB 커밋. 실행 중 SELECT로
  진행 관찰 가능 + 크래시 내성.
- **로컬 검증 250건**: DB반영 249 / 실패 1 (429 백오프가 회복하며 완주), 공실 7건도 0으로 기록.
  이전 코드였다면 ~200건 실패했을 상황.
- 워크플로 env: `SAMSAM_REFRESH_WORKERS=2, SAMSAM_REFRESH_DAILY_LIMIT=5000, SAMSAM_RL_COOLDOWN=20`.
- PR #75 머지 완료. GH Actions run에서 DB 5,300건 갱신 + 스냅샷 2,589건 확인.
