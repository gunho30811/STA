# 2026-07-02

## 삼삼엠투 통합 채팅 (여러 계정 연결)

한 회원이 삼삼엠투 계정을 여러 개 쓸 때, 임대인 모드 채팅(손님 문의)을 한 곳에서 보게 하는 기능.

- **정찰**: 삼삼엠투 채팅은 REST가 아니라 Firebase RTDB(`spacev-33m2-rdb`)를 웹소켓으로 직접 씀.
  로그인 시 33m2 백엔드가 Firebase 커스텀 토큰 발급 → `signInWithCustomToken`으로 교환한 `refreshToken`만
  있으면, 이후엔 브라우저 없이 `securetoken.googleapis.com`(토큰 갱신) + RTDB REST(`live/chatlist/{uid}`,
  `live/messagelist/{room}`)만으로 폴링 가능함을 확인.
- **스키마**(`db.py`): `samsam_accounts`(회원↔삼삼계정, 비번·refreshToken은 `crypto_util.py`(Fernet)로
  암호화 저장) / `samsam_chat_rooms` / `samsam_chat_messages`. 계정당 여러 개 연결 가능하게 정규화 테이블로.
- **`pipeline/samsam/chat_auth.py`**: Playwright 로그인 1회 → refreshToken 획득. 이후 갱신은 순수 HTTP.
- **`pipeline/samsam/chat_poll.py`**: 계정별 폴링. refreshToken 만료 시 저장된 비번으로 자동 재로그인.
  `host_or_guest` 필드로 임대인 모드 채팅만 저장(게스트로 예약한 채팅은 제외).
- **웹**(`web/samsam_app.py` + `templates/samsam_chat.html`): `/samsam/chat/`에서 계정 연결 폼 +
  통합 인박스(방 목록 + 메시지 스레드). "지금 새로고침" 버튼은 그 회원 계정만 즉시 폴링.
- 로컬에서 실계정으로 연결→폴링→UI까지 end-to-end 확인 후, 테스트로 넣었던 계정은 삭제(다른 회원에게
  안 보이게).
- 다음 단계(미착수): 주기적 폴링용 GH Actions workflow.

## 통합채팅 주기적 폴링 workflow + CHAT_ENC_KEY 누락 발견 (feat/samsam-chat-poll-workflow)

- 위 "다음 단계"에 착수하며 `chat_poll.py` 필요 시크릿을 점검하다 **`CHAT_ENC_KEY`가 로컬 `.env`·
  GH secret·`.env.example` 어디에도 없음**을 발견. `SAMSAM_EMAIL/PASSWORD`(크롤러 봇 계정)와 헷갈리기
  쉬운데 별개 용도(계정 비번·refreshToken 암호화)임을 확인.
- DB(`samsam_accounts`) 조회 결과 연결된 계정 0건 — 기존 암호문에 종속된 값이 없어 신규 키 생성이
  안전함을 확인 후 생성 → GH secret 등록. Vercel 프로덕션 쪽은 같은 값을 사용자가 직접 등록.
- `.env.example`에 `CHAT_ENC_KEY` 설명·생성법 추가(재발 방지), 로컬 `.env`에도 반영.
- `pipeline/samsam/decrypt_tool.py` 신설: 암호문 하나를 CLI로 바로 복호화(운영/디버깅용, `chat_poll.py`
  본 폴링 로직과 분리).
- `.github/workflows/samsam-chat-poll.yml` 신설: 10분마다 `chat_poll.py` 실행(+ 수동 실행). 재로그인
  대비 Playwright Chromium도 설치. DB에만 쓰고 커밋할 파일이 없어 `crawl-samsam.yml`과 달리
  PR 자동 생성 단계 없음.
- **프로덕션 실사용 중 발견한 진짜 문제**: 사용자가 실제 계정으로 `/chat/`에서 연결 시도 →
  `ModuleNotFoundError("No module named 'playwright'")`. 원인은 계정 연결(`chat_api_add_account`)이
  로그인을 Vercel 요청 안에서 즉시 처리하는데, Vercel엔 Playwright가 아예 없음(`vercel.json`
  installCommand에서 의도적으로 제외 — 서버리스에 브라우저 자동화 불가). 로컬에서만 테스트해서
  못 잡았던 것으로, **연결 기능이 프로덕션에서 원천적으로 항상 실패**하는 상태였음.
  - 수정: `chat_api_add_account`에서 `ModuleNotFoundError`를 잡으면 즉시 실패 대신 비번만
    암호화해 `status='pending_login'`으로 큐잉하고 안내 메시지 반환.
  - `chat_poll.py`의 `poll_account`를 리팩터: `refresh_token_enc`가 없는(최초 로그인 대기) 계정도
    기존 "재로그인" 경로(비번 기반 로그인)로 통합 처리 — GH Actions(Playwright 설치됨)가 다음
    폴링 주기(10분 이내)에 실제 로그인을 대신 완료.
    프론트(`samsam_chat.html`)에 `pending_login` 상태 배지("연결 처리 중")·안내 메시지 추가.
  - 로컬에서 mock으로 4가지 분기(최초 로그인 성공/실패, 갱신 성공, 갱신 실패→재로그인) 검증.
